Analyst runbook

Run the Hisco rollup end to end.

A browser-openable carousel for this morning's analyst demo: inputs, local config, preflight, run, inspect, SQL check, optional push, and troubleshooting.

Outcome

RiskLink + Verisk YLTs + seeds → Hisco-ready outputs

The pipeline reads analyst-owned data, applies the factor chain in Polars, writes parquets and reports, and can optionally push the Hisco tables into SQL Server.

demo command path
$ cp rollup.example.toml rollup.local.toml $ uv run rollup plan $ uv run rollup run --yes $ uv run rollup test-sql --schema dbo

This is not a code tour — it is the exact path an analyst follows to prove the inputs, run the rollup, and hand off outputs.

Mental model

One local process, typed boundaries.

The tool turns vendor event loss data and run-scope seeds into Hisco parquets, summary reports, debug files when enabled, and an optional SQL Server push.

01

Seeds

Run scope, LOB/peril mappings, analyses, forecast factors, FX, EUWS, and overrides.

02

YLTs

Verisk/AIR and RiskLink/RMS parquet extracts under vendor folders.

03

Polars DAG

Staging, factor attachment, metrics, fanout, and reports in one optimized graph.

04

Files

Hisco parquets plus combined factors, reports, and optional audit parquets.

05

SQL

Probe connection first; push after review only when the analyst confirms.

The important point is control: every source file is accounted for before the expensive run starts.

Data layout

Put the files where the plan expects them.

These folders are analyst-owned. The code stays in polars/; real run data belongs under data/.

repo layout
data/ seeds/ CSV reference inputs: lobs, perils, analyses, factors, fx, EUWS ylt/verisk/ AIR / Verisk YLT parquet extracts ylt/risklink/ RMS / RiskLink YLT parquet extracts ep_summaries/verisk/ optional EP summary review inputs ep_summaries/risklink/ optional EP summary review inputs output/ Hisco parquets, reports, debug audit folders

If the plan fails, start here first: most demo issues are a missing seed, wrong vendor folder, or stale output folder.

04 / 11
Local setup

Copy config once; keep secrets local.

rollup.local.toml is the operator's machine-specific config. It is not committed, and it can hold path overrides, run options, and SQL connection settings.

first-time setup
$ uv sync $ cp rollup.example.toml rollup.local.toml $ $EDITOR rollup.local.toml
SQL optional

Configure only if pushing to SQL Server

Set [sql].mssql_conn_str in rollup.local.toml or provide ROLLUP_MSSQL_CONN_STR. Skip it for parquet-only dry runs and file review.

For the demo, we can run completely file-only, then separately prove whether SQL is reachable.

05 / 11
Preflight

Print the plan before the run.

uv run rollup plan validates the configured inputs and prints what the pipeline will touch without writing the Hisco outputs.

preflight
$ uv run rollup plan [seeds] ✓ required CSVs present and schema-checked [ylt verisk] ✓ data/ylt/verisk/*.parquet [ylt risklink]✓ data/ylt/risklink/*.parquet [outputs] will write Hisco parquets and reports under data/output/ [sql] available only if mssql_conn_str is configured

Pause here in the demo and ask: do the counts, vendor folders, and output path look like the analyst expects?

06 / 11
Run

Run once, then inspect the artifacts.

Use the explicit run subcommand for the demo. Add flags based on the review objective.

run commands
$ uv run rollup run --yes # non-interactive run $ uv run rollup run --yes --no-audit # skip debug audit parquets $ uv run rollup run --yes --min-loss 0 # keep every row $ uv run rollup --log-level INFO run --yes # show factor-chain trace
Default

Production-shaped

Writes outputs with the configured/default minimum loss filter.

No audit

Faster file set

Use when the analyst only needs final Hisco parquets and reports.

Min loss 0

Complete review

Use when reconciling every event row rather than production thresholded rows.

If we have only one live command this morning, make it the preflight, then this run command.

07 / 11
Review

Know what good looks like.

After the run, validate the files by name and purpose before handing them to downstream users.

Hisco parquets

data/output/Hisco*.parquet

Final AIR/RMS fanout tables for main and dials-up variants.

Combined factors

mts_tbl_ylt_combined_all_factors.parquet

Review table with the attached factor chain and traceable intermediate columns.

Audit

data/output/debug/

Debug audit files when audit output is enabled; useful for read-across and pivot review.

Reports

data/output/*report*

Summary/report artifacts written by the reports layer for analyst checks.

Do not push to SQL until the file outputs look right. SQL should be the final delivery step, not the first validation step.

08 / 11
SQL check

Probe SQL before pushing.

The SQL check is read-only. It proves the connection string works and optionally verifies that the target schema exists.

sql probe
$ uv run rollup test-sql --schema dbo SQL Server ✓ connection opened Version reported by @@VERSION Schema ✓ dbo exists
  • If this fails, fix [sql].mssql_conn_str before attempting a push.
  • If the schema check fails, create/confirm the target schema or choose the correct one.

This is a safe checkpoint to run live because it does not write or replace tables.

09 / 11
Optional delivery

Push to SQL only after review.

Current master cleanly skips an interactive SQL push when the analyst declines. Non-interactive shells still require --yes to avoid accidental writes.

sql push
$ uv run rollup push-to-sql --schema dbo # preview, ask y/N $ uv run rollup push-to-sql --schema dbo --yes # confirmed non-interactive push Preview lists target connection, schema, parquets, and target table names Decline skipped SQL push Accept drops/recreates matching Hisco tables in the target schema

The demo language is deliberate: this is optional delivery. Parquets are the reviewable output; SQL is a controlled publish step.

10 / 11
Troubleshooting

Five checks before escalating.

Most failures are local data or configuration issues. Use the preflight error text to locate the failing boundary.

Seeds

Missing or renamed CSVs

Compare data/seeds/ against the data requirements. Schema errors usually name the frame and columns.

YLTs

Wrong folder or glob

Confirm Verisk files are under data/ylt/verisk and RiskLink files under data/ylt/risklink.

Schema drift

Columns or dtypes changed

Vendor extracts and seeds are schema-checked; fix the source export or mapping, then rerun the plan.

SQL

Bad connection string

Run uv run rollup test-sql --schema dbo and verify driver, server, database, credentials, and schema.

Stale output

Old files confuse review

Clear or archive data/output/ and data/output/debug/ before a clean demo rerun.

Flags

Unexpected row counts

Check whether the run used default min-loss filtering, --min-loss 0, or --no-audit.

The fastest recovery path is plan, fix the named input, rerun plan, then rerun the pipeline.

Speaker notes

One-liners for the live demo.

Use these as presenter beats. Keep the demo moving; only drill into code if the analyst asks.

  • "The contract is simple: seeds and vendor YLTs in, Hisco parquets and reports out."
  • "The plan command is our safety gate; it tells us what will happen before we write anything."
  • "The local TOML keeps machine-specific settings and SQL credentials out of git."
  • "Use --min-loss 0 when you want reconciliation breadth rather than production thresholding."
  • "Audit files are for debugging and read-across; --no-audit keeps the file set lean."
  • "Review parquets first, then test SQL, then optionally push."
  • "Declining an interactive SQL push is a clean skip on this branch."
  • "If it fails, trust the boundary message: missing input, schema drift, SQL, or stale files."

Close by opening data/output/ and pointing at the exact files the analyst should hand off or review.
